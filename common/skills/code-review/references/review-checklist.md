# Code Review Checklist

## 目的

代码评审是保障项目代码质量的核心环节，旨在：

1. 保障质量：在代码合并前发现潜在 Bug、性能瓶颈与安全风险。
2. 知识传递：促进团队成员间的设计思路交流与技术方案同步。
3. 维护一致性：确保代码符合技术栈规范与项目架构要求。
4. 规范闭环：验证代码实现与需求场景、变更说明和测试证据一致。

## 评审清单

### 正确性

- [ ] 代码是否完整实现了需求？是否遗漏关键场景？
- [ ] 逻辑是否存在缺陷，例如越界、空指针、资源未释放？
- [ ] 异常处理是否完备？失败路径是否可预期？

### 安全性

- [ ] 是否存在敏感信息泄露、权限绕过或输入校验缺失？
- [ ] 是否对外部输入进行了参数化、白名单或边界校验？
- [ ] 是否遵循最小权限原则？

### 性能

- [ ] 是否存在循环内查库、无界批量、无效对象创建等明显问题？
- [ ] 算法复杂度、缓存策略和并发模型是否合理？
- [ ] 资源使用是否可接受，包括内存、CPU、网络与连接占用？

### 可读性与可维护性

- [ ] 命名是否直观，职责是否清晰？
- [ ] 函数是否过长、嵌套过深或耦合过重？
- [ ] 是否存在重复代码和无价值注释？

### 测试覆盖

- [ ] 是否包含必要的单元测试、切片测试或集成测试？
- [ ] 是否覆盖核心逻辑、异常路径与边界情况？
- [ ] 测试是否独立、可重复运行？

### 边界条件

- [ ] 是否处理了空集合、超时、网络失败、并发竞争等边界情况？
- [ ] 是否识别了回滚、重试、幂等等联动风险？

## PR/MR 描述建议

一个高质量的 PR 描述应包含：

1. 变更摘要
2. 变更类型，如 `feat`、`fix`、`refactor`
3. 关联需求或 Issue
4. 关键改动点
5. 验证步骤
6. 风险说明和回滚关注点

## 反馈分级

| 等级 | 说明 |
| --- | --- |
| 建议 | 可优化项，不阻塞合并 |
| 需要修改 | 明显缺陷、误导性实现或验证不足，修复后再合并 |
| 阻塞 | 重大安全、架构、数据或性能问题，禁止直接合并 |

## 自审流程

1. 再看一遍需求和变更说明，确认没有遗漏场景。
2. 本地运行必要验证，不能只凭推断宣称“应该没问题”。
3. 清理调试代码、临时注释和无关改动。
4. 逐行审视 `git diff`，确认每一行都必要。
5. 为关键结论补充测试日志、截图或命令输出等证据。
